# Provenance Collector docs (Astro + Starlight)

Astro-powered documentation site for `provenance-collector-pack`, built on
Starlight with the shared `@nebari/starlight` theme. Mirrors the pattern used by
`nebari-dev/llm-serving-pack`, with a GitHub Pages target while the Cloudflare
Pages / `packs.nebari.dev` routing is handled in a separate PR.

Deployed at **`https://nebari-dev.github.io/provenance-collector-pack/`**
(GitHub Pages project site).

## Local development

```sh
cd docs
npm install
npm run dev     # http://localhost:4321
```

Then edit files under `src/content/docs/`. Astro reloads on save.

## Layout

Relative to this directory:

- `astro.config.mjs` - Starlight + `@nebari/starlight` + `rehype-mermaid` + `remark-base-links`
- `package.json`, `tsconfig.json`, `.gitignore`
- `src/content.config.ts` - Starlight docs collection
- `src/content/docs/*.md` - actual pages (frontmatter required: `title`, `description`)
- `src/plugins/remark-base-links.ts` - rewrites root-absolute links when deployed under a subpath
- `tests/remark-base-links.test.ts` - vitest for the plugin

Related, at the repo root:

- `.github/workflows/docs.yml` - build, link-check, deploy to GitHub Pages on push-to-main
- `scripts/check-links.sh` - internal-link validator against `docs/dist/`

## Deploy pipeline

`docs.yml` runs on push-to-main and on PRs touching `docs/**`,
`scripts/check-links.sh`, or the workflow itself. Both events run the build
job; only `push` to `main` runs the deploy job.

Build job:

1. Node 22, `npm ci` in `docs/`.
2. Installs Playwright (for `rehype-mermaid`).
3. Runs `vitest` (the `remark-base-links` plugin test).
4. Builds with `SITE=https://nebari-dev.github.io` and `BASE=/provenance-collector-pack/`
   so absolute links resolve correctly under the org-pages subpath.
5. Runs `scripts/check-links.sh` against `docs/dist/` - every internal `href`/`src` must resolve to a file.
6. Uploads `docs/dist/` as the Pages artifact.

Deploy job (main only): `actions/deploy-pages` publishes the artifact.

## Prerequisites (one-time)

Before the first push-to-main deploy will succeed, GitHub Pages needs to be
enabled on the repo:

1. **Settings -> Pages -> Source: GitHub Actions.** The `pages:write` +
   `id-token:write` permissions in the workflow rely on this.
2. On the first successful deploy the URL becomes
   `https://nebari-dev.github.io/provenance-collector-pack/`.

No secrets required beyond the built-in `GITHUB_TOKEN`. No third-party project
setup. Until Pages is enabled, the deploy job fails with a Pages-not-enabled
error; the build + link-check gate still runs on every PR.

Cloudflare Pages + `packs.nebari.dev/provenance-collector-pack/` routing (via
`nebari-dev/software-pack-dashboard`, gated on `docs_site: true` in
`pack-metadata.yaml`) is a deliberate follow-up: once the docs are proven on
GitHub Pages, the same build artifact can be redirected to a Cloudflare Pages
project and the pack registered upstream.
